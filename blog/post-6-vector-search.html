Understanding Vector Embeddings

Vector embeddings are numerical representations of data that capture semantic meaning. Text, images, audio, and other data types can be converted into arrays of floating-point numbers where similar concepts have vectors that are mathematically close to each other in multi-dimensional space. This is the magic behind "king - man + woman ≈ queen" — the model understands relationships, not just co-occurrence.

Classic Vector Arithmetic Example:

"King" - "Man" + "Woman" ≈ "Queen"

Typical vector dimensions: 768, 1024, 1536, or 3072 numbers
Each dimension captures a different aspect of semantic meaning

How Vector Search Works (Step by Step)

  1. Query Embedding — Convert the user's search query into a vector embedding using the same model used for documents.
  2. Similarity Calculation — Compute distance between query vector and stored document vectors using metrics like cosine similarity or Euclidean distance.
  3. Nearest Neighbor Search — Find the K vectors with the highest similarity scores (lowest distance).
  4. Result Retrieval — Fetch the original content (text, image URL, etc.) associated with those vectors.
  5. Optional Reranking — Apply a cross-encoder model for even higher precision on the top candidates.

OCI Vector Search Architecture

Oracle Cloud Infrastructure provides native vector search capabilities that integrate seamlessly with existing Oracle Database workloads:

  • Native Vector Data Type — Storage for vectors up to 3072 dimensions with efficient compression.
  • Similarity Operators — VECTOR_DISTANCE with support for cosine, dot product, Euclidean, and Manhattan distance.
  • Hybrid Search — Combine vector similarity with keyword search (CONTAINS) and JSON filters for precise filtering.
  • Vector Indexes — IVF (Inverted File Index) for approximate nearest neighbor (ANN) search, dramatically speeding up large-scale queries.
  • Partitioning Support — Partition vector tables by metadata (date, category) for even faster searches.
-- SQL Example: Vector Similarity Search with Filtering
SELECT chunk_content, 
       VECTOR_DISTANCE(embedding, :query_embedding, COSINE) as similarity
FROM documents
WHERE category = 'technical_docs'     -- Metadata filter first
  AND VECTOR_DISTANCE(embedding, :query_embedding) < 0.75  -- Threshold
ORDER BY similarity
FETCH FIRST 5 ROWS ONLY;

Choosing the Right Vector Dimensions

The dimension size affects accuracy, speed, and storage. Here's my rule of thumb based on production experience:

768 Dimensions

Faster, less memory
~85-90% accuracy
Best for: real-time search, mobile

1536 Dimensions

Industry standard
~93-96% accuracy
Best for: most production RAG

3072 Dimensions

Most accurate, slower
~97-99% accuracy
Best for: high-precision retrieval

Recommendation: Start with 1536 dimensions (OpenAI's text-embedding-3-small or Cohere embed-english-v3.0). It's the best trade-off for 90% of use cases.

Performance Optimization Strategies

  • Index Selection — Use IVF indexes for datasets with 1M+ vectors. For smaller datasets, brute-force (no index) is actually faster.
  • Approximate Nearest Neighbor (ANN) — Trade ~2-5% accuracy for 10-100x speed improvement. Use for real-time applications.
  • Batch Embedding Generation — Generate embeddings in batches of 100-500 to maximize GPU/API throughput.
  • Dimensionality Reduction (PCA) — Reduce 1536D → 256D with minimal accuracy loss (2-3%) but 6x faster search.
  • Quantization — Convert float32 vectors to int8 or binary for 4x storage reduction.
  • Pre-filtering — Apply metadata filters BEFORE vector search when possible to shrink search space.
